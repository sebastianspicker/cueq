# P5.1: Page Quality & Accessibility Audit

You are auditing the Next.js frontend pages. Work through ONE page at a time.

There are 14 pages under `apps/web/src/app/[locale]/`:
dashboard, bookings, team-calendar, leave, roster, approvals, time-engine,
closing, reports, oncall, policy-admin, audit, settings, (root page.tsx)

Frontend conventions are documented in `docs/FRONTEND.md`.

## What To Do

For each page:

1. Read the page component and verify:
   - Uses the shared API context/client from `src/lib/api-client.ts`
     (no duplicate per-page fetch logic)
   - Uses the `PageShell` component for consistent layout
   - All user-facing text comes from `src/messages/de.json` and `src/messages/en.json`
     (no hardcoded strings)
   - Loading states are handled (LoadingSpinner component)
   - Error states are handled (ErrorBoundary component)
   - Empty states are handled (EmptyState component)

2. Check accessibility:
   - All form inputs have associated labels (htmlFor + id)
   - Buttons have accessible names (not just icons)
   - Tables have proper headers and caption/summary
   - Color is not the sole means of conveying information
   - Focus management for interactive elements

3. Check i18n completeness:
   - Every key used in page components exists in BOTH de.json and en.json
   - No missing translations (search for raw strings that should be translated)
   - Date/number formatting uses locale-aware formatters

4. Fix issues found. Add aria-labels, fix missing translations, extract hardcoded strings.

## What NOT To Do

- Do NOT modify API services or core domain logic
- Do NOT add new pages or routes
- Do NOT change the routing structure
- Do NOT install new npm packages
