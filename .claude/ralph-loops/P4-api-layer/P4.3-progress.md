### Bug Fix: Closing Helper Transition Violations Error Shape

**Found in 4 files**: `closing-lifecycle.helper.ts` (3 locations), `closing-export.helper.ts` (1),
`closing-correction.helper.ts` (1) — `transition.violations` (string[]) was passed directly
to `BadRequestException`, producing a raw JSON array response body instead of the expected
`{ statusCode, error, message, details }` object shape.

**Fixed**: All 4 locations now return structured `{ statusCode: 400, error: 'Bad Request', message, details }`,
matching the pattern already used in `workflow-runtime.service.ts:230-235`.

## Audit 2: Exception Filters

| Category             | Status        | Notes                                                |
| -------------------- | ------------- | ---------------------------------------------------- |
| Error response shape | ✅ Consistent | message: string, details: string[] (when applicable) |
| Correlation IDs      | ⚠️ Gap noted  | Neither filter includes request correlation IDs      |
| Stack trace leaking  | ✅ Safe       | Verified via unit + integration tests                |
| PII in errors        | ✅ Clean      | Generic domain terms only, verified in tests         |

**Correlation ID gap**: No request/correlation ID in error responses. This requires request-scoped
middleware to generate and propagate IDs — a feature addition, not a bug. Noted for future work.

**Response shape consistency**: PrismaExceptionFilter returns `{statusCode, error, message}` while
ZodExceptionFilter/ZodValidationPipe include `details: string[]`. The `details` field is
validation-specific, so this variation is intentional — database errors don't have structured
details to expose.
