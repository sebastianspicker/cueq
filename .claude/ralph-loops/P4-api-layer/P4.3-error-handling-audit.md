# P4.3: Error Handling & Observability Audit

You are auditing error handling patterns across the API layer.
Work through items ONE AT A TIME.

## What To Do

1. Check that all service methods have proper error handling:
   - Prisma operations wrapped with typed error translation (not bare try/catch)
   - Domain errors (from `packages/core/`) propagated with correct HTTP status codes
   - 404 for not found, 400 for validation, 403 for role violations, 409 for conflicts
   - No swallowed exceptions (catch without re-throw or logging)

2. Check the ZodExceptionFilter and any global exception filters:
   - Do they return consistent error response shapes?
   - Do they include correlation IDs?
   - Do they NOT leak stack traces in production?
   - Do they NOT leak PII in error messages?

3. Check logging patterns:
   - Are security-relevant events logged (login, role change, export)?
   - Do logs avoid PII (no names, emails, or tokens in log output)?
   - Is the logging level appropriate (info vs debug vs warn)?

4. Check integration test coverage for error paths:
   - `apps/api/test/` — do tests exercise 4xx and 5xx paths?
   - Are there tests for concurrent modification conflicts?
   - Are there tests for missing referenced entities?

5. Add missing error-path tests where gaps exist

## What NOT To Do

- Do NOT modify core domain logic
- Do NOT change schemas or frontend code
- Do NOT add rate limiting (documented as accepted risk)
- Focus on error handling, not new features
