### Integration Endpoints (Service-Level Validation)

Three controllers use @Public() with integration token auth. Their Zod validation happens in the service layer via .parse():

- terminal-sync.controller.ts: TerminalSyncBatchSchema/TerminalSyncBatchFileSchema in terminal-gateway.service.ts
- terminal-integration.controller.ts: TerminalHeartbeatSchema in terminal-gateway.service.ts
- hr-import.controller.ts: HrImportPayloadSchema in hr-import.service.ts

These schemas are defined locally in services, not in @cueq/shared. The global ZodExceptionFilter catches validation failures and returns structured errors.

## Pass 3 — Independent Verification Audit

Full re-read of all 18 controllers and all shared schemas. Findings:

### Issue Found & Fixed

| Controller    | Endpoint   | Issue                                                      | Fix                                                  |
| ------------- | ---------- | ---------------------------------------------------------- | ---------------------------------------------------- |
| terminal-sync | POST /     | Raw @Body() payload: unknown — service-level .parse() only | Added ZodValidationPipe(TerminalSyncBatchSchema)     |

Exported TerminalSyncBatchSchema and TerminalSyncBatchFileSchema from terminal-gateway.service.ts so they can be referenced at the controller level. Service-level .parse() still runs as defense-in-depth.

### Noted but Not Fixed (Architectural)

- **correlationId**: The ApiErrorSchema in common.ts defines an optional correlationId field, but no middleware/interceptor generates or injects it. This is an architectural concern that would require a correlation ID middleware (e.g., attaching a UUID to each request via cls-hooked or AsyncLocalStorage) — out of scope for controller-level audit.
