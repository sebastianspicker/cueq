## Documented Mismatches (Intentional / Out of Scope)

### 1. ClosingStatus Enum — APPROVED vs CLOSED (INTENTIONAL)

- **Prisma**: `ClosingStatus { OPEN, REVIEW, CLOSED, EXPORTED }`
- **Zod/JSON core**: uses `APPROVED` instead of `CLOSED`
- **Reason**: Intentional domain design. The core domain layer uses `APPROVED`
  as a business concept while Prisma uses `CLOSED`. Mapping functions exist in
  `closing-lock.helper.ts` and `closing-domain.service.ts`.

### 2. Person Schema — Missing Employment Date Fields

- **Missing from JSON**: `employmentStartDate`, `employmentEndDate`
- **Present in Prisma**: Both as `DateTime?`
- **Not fixed**: Per "do NOT add new fields" rule

### 3. ExportRun Schema — Missing Artifact Fields

- **Missing from JSON**: `artifact` (String?), `contentType` (String?)
- **Not fixed**: Per "do NOT add new fields" rule

### 4. Absence startDate/endDate — Date vs DateTime

- **Prisma/JSON**: `DateTime`/`dateTime`
- **Zod**: `DateSchema` (date-only `YYYY-MM-DD`)
- **Not fixed**: Semantically correct for absence dates

### 5. TerminalHeartbeat — Input vs Stored Schema

- JSON schema is for input validation, Prisma model is for stored records
- **Not fixed**: Intentional design

### 6. OnCallDeployment endTime Nullability (Zod only)

- **Prisma**: required, **Zod**: nullable
- **Not fixed**: No JSON schema, out of scope
