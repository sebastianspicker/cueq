# P6.3 Test Quality & Reliability Audit — Progress

## Audit Summary

Audited **44 unit test files** across all 6 packages (packages/core, packages/policy,
packages/shared, packages/database, apps/api, apps/web). Identified and fixed issues
in 6 categories: flaky patterns, isolation, assertion quality, naming, structure, and performance.

## Item 1: Flaky Test Patterns — Fixed

### utils.test.ts — wall-clock dependency ✅

- `toIso()` no-arg call depended on `new Date()` and asserted with a weak regex
- **Fix**: Added `vi.useFakeTimers({ now: ... })` and asserted exact ISO string

### workflow.test.ts — wall-clock dependency ✅

- `transitionWorkflow` with omitted `at` generated `decidedAt` from wall-clock
- Asserted via weak regex `toMatch(/^\d{4}-\d{2}-\d{2}T/)`
- **Fix**: Added `vi.useFakeTimers` and asserted exact `decidedAt` value

### build-audit-entry.test.ts — weak timestamp regex ✅

- `entry.timestamp` asserted with incomplete regex `toMatch(/^\d{4}-\d{2}-\d{2}T/)`
- **Fix**: Strengthened to full ISO 8601 pattern

### Other patterns checked (no issues found)

- No `setTimeout`/`setInterval` in any unit tests
- No `Date.now()` usage in any unit tests
- No execution order dependencies between tests
- Shared `BASE_INPUT` objects in flextime/oncall-rest/evaluate-time-rules tests
  are always spread-copied — no mutation risk
- `new Date()` only appears in integration tests (excluded from unit run)

## Item 2: Test Isolation — Fixed

### api-client.test.ts — global fetch mutation ✅

- Directly assigned `globalThis.fetch = fetchMock` and manually restored via `ORIGINAL_FETCH`
- If test threw before `afterEach`, fetch stayed corrupted for subsequent tests
- **Fix**: Replaced with `vi.spyOn(globalThis, 'fetch').mockResolvedValue(...)` + `vi.restoreAllMocks()`

### Other isolation patterns checked (acceptable)

- `integration-token.test.ts` and `auth.service.test.ts` use manual `restoreEnv()` — safe
- `webhook-url.test.ts` has `afterEach` at module level — works in vitest
- `process.env` mutations in unit tests are properly restored via `afterEach` cleanup

## Item 3: Assertion Quality — Fixed

### zod-validation.pipe.test.ts — anti-pattern ✅

- Used `try/catch` + `expect.fail('Should have thrown')` instead of `.toThrow()`
- **Fix**: Replaced with `toThrow(BadRequestException)` guard + specific assertion on
  `response.details` checking exact messages

### utils.test.ts — weak assertion on overlap index ✅

- `expect(issues[0]?.index).toBeDefined()` only checked existence, not the specific value
- **Fix**: Strengthened to `expect(issues[0]?.index).toBe(1)` — the interval at original
  index 1 (08:00-13:00) is sorted first and detected as the overlap source

### closing.test.ts — non-null assertions after find() ✅

- 6 instances of `checklist.items.find(...)!.severity` used TypeScript non-null assertion (`!`)
- If `find()` returned undefined, tests would throw `TypeError` instead of a meaningful
  assertion failure
- **Fix**: Replaced `!` with optional chaining (`?.`) — on failure, Vitest now reports
  `expected undefined to be 'INFO'` instead of a cryptic TypeError

### Other assertion quality checked (acceptable)

- evaluate-time-rules.test.ts and flextime.test.ts use `toBeDefined()` + `!` as a
  two-assertion guard pattern — the `toBeDefined()` provides the meaningful error,
  and the `!` is safe behind it
- UUID regex, error assertions in absence.test.ts, mock call assertions all adequate

## Item 4: Test Naming — Fixed

### zod-validation.pipe.test.ts — vague names ✅

- `describe('with simple schema')` → `describe('with scalar schema')`
- `it('works with string schema')` → `it('validates and transforms non-object schemas')`

### Other naming checked (acceptable)

- Core tests follow `describe [module] > it [behavior when condition]` consistently
- New test files (break-utils, parse-cuid, roles.guard, mock-identity-provider) all
  follow naming conventions

## Item 5: Test Structure — Fixed

### fixture-parity.test.ts — monolithic test ✅

- Single `it()` block ran 6 fixture files in a for-loop with if/continue chains
- **Fix**: Split into 6 individual `it()` blocks, one per fixture

## Item 6: Performance — No Issues

- 560 tests across 44 files in 6 packages
- No accidentally slow tests found (no unnecessary async, no large data sets)
