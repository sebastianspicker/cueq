# P6.1: Test Coverage Gap Analysis

You are analyzing test coverage across the entire cueq monorepo.
Work through items ONE AT A TIME.

Quality targets from `docs/QUALITY_SCORE.md`:

- `packages/core/src/core/` — >=90% line coverage
- Adapters — >=70% line coverage
- API endpoint coverage — 100% of OpenAPI spec
- Component coverage — >=60%
- Overall — >=80%

Test suite structure:

- Unit: `*.test.ts` files co-located with source or in `__tests__/`
- Integration: `test/integration/` in apps/api
- Acceptance: `test/acceptance/` in apps/api, `*.acceptance.test.ts` in core
- Compliance: `test/compliance/` in apps/api, `*.compliance.test.ts` in packages
- Smoke: `*.smoke.test.ts`

## What To Do

1. For each package/app, list all source files and their corresponding test files
2. Identify source files with NO corresponding tests:
   - Each file in `packages/core/src/core/` — has tests?
   - Each `apps/api/src/phase2/services/*.service.ts` — has integration tests?
   - Each `apps/api/src/phase2/controllers/*.controller.ts` — has contract tests?
   - `apps/api/src/common/` files beyond the ones with existing tests

3. For files that have tests, check if the tests are substantive:
   - Are there just smoke tests (checking it "doesn't throw")?
   - Do tests cover error paths, not just happy paths?
   - Do acceptance tests actually assert on response data?

4. Create a coverage gap report in `.claude/ralph-loops/P6-testing/coverage-gaps.md`
5. Write the highest-priority missing tests (prioritize core domain > API > frontend)

- Do NOT change schemas or configs
- Do NOT add integration tests that require Docker (save for P6.2)
