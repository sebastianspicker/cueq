# P6.3: Test Quality & Reliability Audit

You are auditing test quality — not coverage (P6.1/P6.2 handled that) but
whether the tests are well-written, reliable, and maintainable.
Work through items ONE AT A TIME.

## What To Do

1. Check for flaky test patterns:
   - Tests that depend on wall-clock time (use deterministic dates from fixtures)
   - Tests that depend on execution order
   - Tests with `setTimeout` or polling
   - Tests that share mutable state between cases

2. Check test isolation:
   - Each test should set up its own data (no reliance on other tests)
   - Database tests should use transactions or cleanup
   - Mock state should be reset between tests

3. Check assertion quality:
   - Tests should assert specific values, not just "truthy"
   - Error case tests should assert specific error types/messages
   - Snapshot tests (if any) should be small and focused

4. Check test naming:
   - Test names should describe the behavior being tested
   - Not "test1", "works correctly", "should work"
   - Should follow pattern: "describe [module] > it [behavior when condition]"

5. Check test performance:
   - Unit tests must complete in <10s total (per QUALITY_SCORE.md)
   - Look for tests that are accidentally slow (unnecessary async, large data sets)

6. Fix issues found

## What NOT To Do

- Do NOT modify source code
- Do NOT change schemas or configs
- Do NOT add new features
